À propos du Projet NASA

Une exploration des données spatiales à travers une application web dynamique

Présentation du Projet

Ce projet est une application web dédiée à l'exploration des données de la NASA. Notre application permet aux utilisateurs de découvrir et d'explorer des images, des catégories et des ressources liées à l'espace et aux missions spatiales.

Fonctionnalités principales

  • Navigation par catégories de contenu spatial
  • Recherche de ressources et d'images
  • Visualisation détaillée des collections
  • Accès aux informations sur les missions et projets de la NASA

Installation et démarrage

  1. Clonez le dépôt GitHub :
    git clone https://github.com/votre-compte/projet-nasa.git
  2. Accédez au répertoire du projet :
    cd projet-nasa
  3. Installez les dépendances :
    go mod download
  4. Démarrez le serveur :
    go run main.go
  5. Accédez à l'application dans votre navigateur :
    http://localhost:8080

Routes implémentées

Route Méthode Description
/ GET Page d'accueil
/categories GET Liste de toutes les catégories
/category/:id GET Détails d'une catégorie spécifique
/collection/:id GET Affichage d'une collection spécifique
/resource/:id GET Détails d'une ressource spécifique
/search GET Recherche de ressources
/about GET Page à propos du projet

API NASA utilisée

Ce projet utilise l'API publique de la NASA pour récupérer des données authentiques sur l'espace et les missions spatiales.

  • Base URL: https://api.nasa.gov/ https://images-api.nasa.gov/ https://gibs.earthdata.nasa.gov/
  • Endpoints utilisés:
    • /planetary/apod - Image astronomique du jour
    • /mars-photos/api/v1/rovers/{rover}/photos - Photos des rovers martiens
    • /search (sur images-api.nasa.gov) - Recherche dans la bibliothèque d'images NASA
    • /wmts/epsg4326/best/MODIS_Terra_CorrectedReflectance_TrueColor/default/{date}/250m/{zoom}/{y}/{x}.jpg - Images satellite de la Terre
    • https://api.rss2json.com/v1/api.json - Conversion du flux RSS des actualités NASA
    • https://api.mapbox.com/styles/v1/mapbox/satellite-v9/static/{lon},{lat},{zoom},0/600x400 - Images satellite Mapbox
    • https://maps.googleapis.com/maps/api/staticmap - Solution de secours pour images satellite

FAQ - Déroulement du Projet

Comment avons-nous décomposé le projet ? Quelles ont été les phases clé ?

Notre approche de décomposition du projet s'est articulée autour des phases suivantes :

  1. Phase d'analyse (1 semaine) : Nous avons commencé par étudier l'API de la NASA pour comprendre les données disponibles et définir le périmètre du projet.
  2. Phase de conception (1 semaine) : Nous avons conçu l'architecture MVC (Model-View-Controller) de notre application en Go, défini les structures de données et les modèles de pages.
  3. Phase de développement (3 semaines) :
    • Développement du backend en Go avec création des contrôleurs et des routes
    • Intégration avec l'API NASA
    • Création des templates HTML et des styles CSS
  4. Phase de test et d'optimisation (1 semaine) : Tests des fonctionnalités, correction des bugs et optimisation des performances.
  5. Phase de finalisation (1 semaine) : Documentation, déploiement et préparation de la présentation.

La décomposition modulaire nous a permis de travailler en parallèle sur différentes parties du projet, facilitant l'intégration continue.

Comment avons-nous réparti les tâches entre les membres de l'équipe ? Avons-nous utilisé une stratégie particulière pour organiser le travail ?

Nous avons adopté une approche basée sur les compétences et les préférences de chaque membre :

  • Organisation par domaines de compétence : Chaque membre a pris en charge un aspect du projet correspondant à ses forces :
    • Membre 1 : Backend et contrôleurs Go
    • Membre 2 : Templates HTML et intégration frontend
    • Membre 3 : Styles CSS et design responsive
    • Membre 4 : Intégration API et gestion des données
  • Méthode agile adaptée : Nous avons utilisé une version simplifiée de Scrum avec :
    • Sprints hebdomadaires
    • Réunions quotidiennes de 15 minutes pour partager les avancées et les blocages
    • Tableau Kanban (via GitHub Projects) pour visualiser les tâches en cours, à faire et terminées
  • Revue de code croisée : Chaque pull request devait être validée par au moins un autre membre de l'équipe avant d'être fusionnée dans la branche principale.

Cette organisation nous a permis de maintenir un rythme de développement constant tout en assurant la qualité du code.

Comment avons-nous géré notre temps ? Avons-nous défini des priorités ?

Notre gestion du temps s'est appuyée sur plusieurs principes :

  • Matrice MoSCoW pour les priorités :
    • Must have : Fonctionnalités essentielles (navigation par catégories, affichage des ressources)
    • Should have : Fonctionnalités importantes mais non critiques (recherche avancée, filtres)
    • Could have : Améliorations souhaitables (animations, fonctionnalités supplémentaires)
    • Won't have : Fonctionnalités reportées à une version ultérieure
  • Planning par jalons : Nous avons défini des jalons clairs à atteindre toutes les deux semaines, avec des démos intermédiaires.
  • Gestion des risques : Nous avons identifié les zones à risque (notamment l'intégration avec l'API NASA) et les avons abordées en priorité.
  • Marges de sécurité : Nous avons volontairement prévu une semaine tampon à la fin du projet pour gérer les imprévus et finaliser les éléments en retard.

Cette approche nous a permis de livrer toutes les fonctionnalités essentielles dans les délais, même lorsque nous avons rencontré des difficultés techniques avec certaines parties de l'API.

Quelle stratégie avons-nous adoptée pour nous documenter ?

Notre stratégie de documentation a été multidimensionnelle :

  • Documentation de l'API NASA : Nous avons commencé par étudier la documentation officielle de l'API NASA et créé un document de synthèse des endpoints utiles.
  • Documentation du code :
    • Commentaires en Go pour les fonctions et méthodes complexes
    • Documentation des structures de données et des modèles
    • Description des contrôleurs et de leur rôle
  • Documentation pour les utilisateurs :
    • Création d'un README.md complet avec instructions d'installation et d'utilisation
    • Description des fonctionnalités accessibles
  • Documentation du processus :
    • Journal de bord du projet avec les décisions importantes
    • Documentation des problèmes rencontrés et des solutions apportées
    • Cette page FAQ qui résume notre approche du projet

Nous avons adopté le principe "Documentation as Code" en maintenant nos documents à jour en même temps que le code, plutôt que de tout documenter à la fin. Cette approche a facilité la montée en compétence de tous les membres sur les différentes parties du projet.
